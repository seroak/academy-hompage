# 개발 워크플로 가이드

## 워크트리 생성

새 작업용 워크트리는 프로젝트 루트에서 다음 명령으로만 만든다.

```bash
./scripts/create-worktree.sh <작업명>
```

- 작업명 `feature-name`은 `.worktrees/feature-name` 경로와 `codex/feature-name` 브랜치에 사용된다.
- 명령은 백엔드·프론트엔드 `.env`를 메인 체크아웃의 파일에 심볼릭 링크하고, 양쪽 `npm ci`와 백엔드 Prisma Client 생성을 실행한다.
- 메인 체크아웃에 `backend/.env` 또는 `frontend/.env`가 없으면 생성 전에 중단한다. 시크릿을 임의로 만들거나 출력하지 않는다.
- 수동 `git worktree add`는 환경 파일과 생성 의존성이 누락되므로 사용하지 않는다.

- 작업 시작 전 `git status`, 관련 파일, 최근 `git log --oneline`과 브랜치를 확인한다. 다른 세션·브랜치의 진행 중 작업과 충돌하지 않는다.
- `git add`, `git commit`, 병합, 푸시는 사용자가 해당 동작을 명시적으로 요청한 뒤에만 수행한다. 설계·계획·구현 승인과 도구·샌드박스 승인은 Git 작업 승인으로 간주하지 않는다.
- 백엔드 기능은 `*.spec.ts`를 먼저 작성해 RED를 확인한 뒤 최소 구현으로 GREEN을 만든다.
- lint·정적 분석 경고를 해결할 때 임계값 상향, 규칙 비활성화, 예외 추가는 사용자 승인 없이 사용하지 않는다. `.lighthouserc.cjs`의 성능 판정 기준(수치 임계값, 측정 방식 등) 변경도 동일하게 사용자 승인을 받는다.
- 커밋·검증(빌드, E2E) 직전에 `git status`로 이번 작업과 무관한 변경이 섞여 있지 않은지 재확인한다 — 워크트리를 병행 사용하는 구조라 다른 세션의 진행 중 변경이 섞이기 쉽다.
- `## Page Feedback: /경로` 형식(뷰포트, 컴포넌트 경로, React 트리, 피드백)으로 들어온 요청은 브레인스토밍 설계 문서 절차를 거치지 않고 바로 코드 조사·구현에 착수한다.
- 코드 리뷰에서 발견한 중요한 문제는 완료 전에 수정한다.
- 이 저장소에는 독립 프로젝트가 여럿 있으므로 커밋에 `academy-hompage/` 밖 변경을 섞지 않는다.
- 세션 시작 전부터 있던 untracked 파일은 `git add`에 자동 포함하지 않는다.
- 장기 작업 중에도 `main`은 빌드 가능한 상태를 유지한다. 서버를 종료했다면 완료 보고에 명시한다.
- 컨텍스트 재개 후에는 `HANDOFF.md`와 `docs/solutions/`를 먼저 읽고, 핵심 파일은 다시 확인한다. 장기 다단계 작업은 다음 세션이 이어갈 수 있게 짧은 진행 상태를 남긴다.
- 아키텍처 규칙의 같은 주제가 세 항목 이상으로 늘면 이 디렉터리의 관련 가이드에 하위 섹션으로 분리한다.
- 진단·모니터링용 쉘 스크립트에서 변수명으로 `status`를 쓰면 zsh 예약 변수와 충돌해 스크립트가 exit 1로 조용히 실패할 수 있다 — `status` 대신 `result`, `state` 등 다른 이름을 쓴다.
- 워크트리를 여러 개 병행 사용할 때 포트 점유 프로세스가 어느 워크트리·어느 모드(watch dev vs. 정적 빌드)인지 헷갈리기 쉽다 — `lsof -i :포트`로 PID를 특정한 뒤 `lsof -p <PID> | grep cwd` 또는 `ps -p <PID> -o command`로 실행 디렉터리·명령을 확인해 구분한다.
